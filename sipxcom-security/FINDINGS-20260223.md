## Executive Summary

The server has three working layers of SIP attack protection (APIBAN, fail2ban, manual iptables blocks). The protection is functioning correctly and detecting/banning attackers. However, fail2ban is configured to use raw iptables rules rather than ipset for storing bans. With 5,926 permanently banned IPs creating 5,926 individual iptables rules, the total ruleset has grown to **7,713 rules**. Linux checks these rules sequentially, causing CPU spikes during attack bursts that delay legitimate SIP REGISTER responses — phones time out, lose registration, and go offline. A server reboot clears the accumulated bans and restores normal operation for ~3 days until the rule count builds back up.

**This is a fixable configuration issue, not a fundamental design flaw.** The architecture is correct. Switching fail2ban to ipset resolves the root cause.

## Root Cause Analysis: Registration Loss

### The Problem — Linear iptables Rule Traversal

```
Total iptables rules: 7,713

Breakdown:
  fail2ban sip-dos permanent bans:  ~5,926 rules
  APIBAN cloud blocklist:            1,673 rules
  Manual blocks + other:              ~114 rules
  ─────────────────────────────────────────────
  Total:                             7,713 rules
```

The Linux kernel processes iptables rules **sequentially** (linear search). Every incoming packet — including legitimate SIP REGISTER packets from the Polycom phones — must traverse up to 7,713 rule checks before being accepted.

### The Failure Sequence

1. Attack burst arrives (automated SIP scanners sending REGISTER/INVITE floods)
2. Each attack packet triggers a 7,713-step iptables rule walk in kernel
3. New unique attacker IPs trigger fail2ban to add more iptables rules
4. CPU spikes — kernel is busy processing both the attack traffic and the iptables rule walks
5. Legitimate REGISTER refresh packets from phones queue behind attack traffic
6. Phones don't receive 200 OK before their registration timer expires
7. Phones go offline — no inbound or outbound calls
8. **Reboot clears fail2ban's in-memory bans** — rule count drops to ~1,787 (APIBAN + manual)
9. Normal operation resumes... until bans accumulate back to ~7,000+ rules in 2-3 days

### Reboot History Confirms the Pattern

| Reboot | Date | Uptime Before | Notes |
|--------|------|---------------|-------|
| Stable | Jan 23 – Feb 17 | **24 days** | Rule count was manageable |
| Reboot | Feb 17 05:59 | 24 days | Warren reported registration loss |
| Reboot | Feb 20 09:43 | 3 days | Bans accumulated, failure repeated |
| Reboot | Feb 23 08:24 | 3 days | Same cycle, today's reboot |

The ~3-day cycle began around Feb 17, suggesting attack volume increased at that time and the rule accumulation rate sped up proportionally.

### Additional Contributing Factors

**No swap space:** Server has 0 bytes of swap. RAM is 4GB with 1.1GB free under normal load. During attack bursts, if RAM spikes (connection tracking tables, fail2ban processing), there's no overflow buffer. Any process that runs out of memory is OOM-killed immediately.

**Active SYN flood on port 443:** Today's messages log shows 9 SYN flood alerts between 18:58–20:16. While SYN cookies are handling it (kernel is sending cookies), this still consumes CPU and competes with SIP traffic.

**SIP registrar transaction backlog:** `sipregistrar.log` shows continuous `removeOldTransactions` entries — normal cleanup, but during CPU spikes these maintenance tasks add to processing load.
